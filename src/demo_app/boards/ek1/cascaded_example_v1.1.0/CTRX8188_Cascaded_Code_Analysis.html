Infineon 3x CTRX8188 Cascaded Radar Integration Analysis

Project: MIMOrad - P2S40_3xCTRX8188 Cascaded Example v1.1.0

Platform: NVIDIA Jetson Orin AGX Developer Kit

Radar Board: Infineon P2S40 with 3x CTRX8188 MMIC Radar Chips

Firmware Version: 1.1.0-RamFw_1.0.0-rc3

Configuration: 1 Primary + 2 Secondary Devices in Cascaded Mode

Analysis Date: July 15, 2026

1. High-Level Overview

This codebase implements a cascaded radar system that initializes and operates three Infineon CTRX8188 radar chips synchronously (CTRX_A, CTRX_B, and CTRX_C). The primary purpose is to:

  • Initialize the hardware platform - Configure GPIO, SPI, and I2C interfaces on the NVIDIA Jetson Orin for three radar chips
  • Power up the radar system - Enable two PMICs (Power Management ICs) with spread spectrum control for all three chips
  • Configure radar chips in cascaded mode - CTRX_A as PRIMARY master, CTRX_B and CTRX_C as SECONDARY slaves synchronized via LOIN signals
  • Load chip-specific sequencer programs - Different programs for primary vs secondary chips to coordinate synchronous operation
  • Execute synchronized radar operations - Perform chirp sequences across all three chips simultaneously and stream radar data via three MIPI CSI-2 interfaces
  • Provide user interaction - Allow manual control of synchronized radar operation cycles through console input
Cascaded Mode Architecture: In cascaded mode, CTRX_A acts as the PRIMARY device that generates timing signals. CTRX_B and CTRX_C act as SECONDARY devices synchronized to CTRX_A via LOIN (Local Oscillator Input) signals. This enables coherent MIMO radar operation with synchronized transmit and receive across all 24 RX channels (8 per chip) and up to 12 TX channels (4 per chip).
Data Streaming: Each radar chip streams data through its own dedicated MIPI CSI-2 interface (4 lanes @ 1200 Mbps per chip) to the Jetson Orin's camera subsystem. The three data streams are captured simultaneously as /dev/video0 (CTRX_A), /dev/video1 (CTRX_B), and /dev/video2 (CTRX_C).

2. Step-by-Step Execution Flow

Step 1: Application Initialization

The program starts at main() in main.c:44 and prints a banner message.

printf(" === IfxRfe App - p2s40_3x8188_cascaded_example ===\n");

Three device suffixes are defined for logging: 'A', 'B', 'C' (main.c:46)

Step 2: PMIC Configuration Objects

Two MAX20434 PMIC structures are created (main.c:50-51):

  • pmic_ab (I2C address 0x3E) - Powers CTRX_A and CTRX_B
  • pmic_c (I2C address 0x3F) - Powers CTRX_C

Step 3: GPIO Configuration for All Three Chips (main.c:56-79)

Configure all necessary GPIO pins for three-chip operation using PlatformGpio_configure():

General GPIOs (Shared):

  • GPIO_ID_EN_PMIC - PMIC enable (output, initially LOW)
  • GPIO_ID_CTRX_RESET_N - Radar chip reset for all chips (output, initially LOW)
  • GPIO_ID_CTRX_OK - Radar chip status indicator shared by all chips (input)
  • GPIO_ID_PGOOD_PMIC_LV/HV - PMIC power good signals (inputs)

CTRX_A Specific GPIOs (Primary Device):

  • GPIO_ID_RFT_A - RFT signal (input)
  • GPIO_ID_SPI_SS_A - SPI chip select (output, HIGH when inactive)
  • GPIO_ID_DMUX_1_A - Data multiplexer 1 for synchronization (input)
  • GPIO_ID_DMUX_2_A - Data multiplexer 2 (input)

CTRX_B Specific GPIOs (Secondary Device):

  • GPIO_ID_RFT_B - RFT signal (input)
  • GPIO_ID_SPI_SS_B - SPI chip select (output, HIGH)
  • GPIO_ID_DMUX_1_B - Data multiplexer 1 (input)
  • GPIO_ID_DMUX_2_B - Data multiplexer 2 (input)

CTRX_C Specific GPIOs (Secondary Device):

  • GPIO_ID_RFT_C - RFT signal (input)
  • GPIO_ID_SPI_SS_C - SPI chip select (output, HIGH)
  • GPIO_ID_DMUX_1_C - Data multiplexer 1 (input)
  • GPIO_ID_DMUX_2_C - Data multiplexer 2 (input)

Step 4: I2C Initialization (main.c:82)

Initialize I2C bus ID 2 (I2C3 in hardware) for PMIC communication:

PlatformI2c_init(I2C_ID);

This I2C bus connects to both PMIC devices at addresses 0x3E and 0x3F.

Step 5: SPI Configuration for Three Chips (main.c:86-98)

Configure SPI for communication with all three CTRX chips:

  • Device IDs: 0, 1, 2 (internal IfxRfe device indices)
  • SPI IDs:
    • CTRX_A: CTRX_A_SPI_ID (0x10)
    • CTRX_B: CTRX_B_SPI_ID (0x11)
    • CTRX_C: CTRX_C_SPI_ID (0x12)
  • CS GPIO IDs: GPIO_ID_SPI_SS_A, GPIO_ID_SPI_SS_B, GPIO_ID_SPI_SS_C
PlatformSpi_init(); PlatformSpi_setDevIdSpiIdMapping(devIds, spiIds, sizeof(devIds)); PlatformSpi_setCsGpioIds(spiIds, spiCsGpioIds, sizeof(spiIds));

Note: All three chips share common SPI CLK, MOSI, and MISO lines. Individual chip selection is managed via separate CS GPIO pins.

Step 6: IfxRfe Middleware Initialization (main.c:103-159)

Initialize the Infineon Radio Frequency Engine (IfxRfe) middleware with platform abstraction callbacks:

SPI Functions with Board-Specific Overrides:

IfxRfe_spiFunctions_t spiFncs = { .spiConfigure = PlatformSpi_configure, .spiTransfer = PlatformSpi_transfer_override, // Custom wrapper with delays .spiWrite = PlatformSpi_write_override // Custom wrapper };

The override functions (defined in 3xCtrx8188Wrapper.c) add 2ms delays after reset/LBIST/finish_fw_download commands to allow RFT pin stabilization.

GPIO, Time, and Logging Callbacks:

  • GPIO Functions: get, set, configure, isConfigOutput
  • Time Functions: getDeadLine, now, waitTime
  • Log Callbacks: error, warning, info

GPIO Definitions for Each CTRX Device:

Three IfxRfe_gpioDefinitions_t structures define GPIO mappings (main.c:126-148):

  • CTRX_A: RFT_A, CTRX_OK (shared), DMUX_1_A
  • CTRX_B: RFT_B, CTRX_OK (shared), DMUX_1_B
  • CTRX_C: RFT_C, CTRX_OK (shared), DMUX_1_C
EXIT_ON_IFXRFE_ERROR( IfxRfe_init( CTRX_DEV_COUNT, // 3 devices gpios, spiFncs, gpioFncs, timeFncs, logInterface), cleanup_platform() );

Step 7: Power-Up Sequence (main.c:162-188)

Critical Timing: CTRXs must be held in reset during PMIC power-up to avoid GPIO leakage issues.

7.1 Assert Reset (main.c:163-164)

PlatformGpio_set(GPIO_ID_CTRX_RESET_N, false); // Assert reset (active LOW) PlatformTime_waitTime(10); // Wait 10ms

7.2 Enable PMICs (main.c:166-168)

PlatformGpio_set(GPIO_ID_EN_PMIC, true); PlatformTime_waitTime(10);

7.3 Configure Spread Spectrum (main.c:170-174)

Enable ±3% pseudo-random spread spectrum on both PMICs for EMI reduction:

Max20434_enableSpreadSpectrum(&pmic_ab, SSE_3Percent_PseudoRandom); Max20434_enableSpreadSpectrum(&pmic_c, SSE_3Percent_PseudoRandom);

7.4 PMIC Output Reset Workaround (main.c:176-184)

Hardware Workaround: Due to Jetson GPIO leakage when first connected, PMIC outputs don't power up correctly. The workaround disables and re-enables outputs to reset internal state.
Max20434_setManualOutputEnable(&pmic_ab, false); Max20434_setManualOutputEnable(&pmic_c, false); PlatformTime_waitTime(100); Max20434_setManualOutputEnable(&pmic_ab, true); Max20434_setManualOutputEnable(&pmic_c, true); PlatformTime_waitTime(10);

7.5 De-assert Reset (main.c:186-188)

PlatformGpio_set(GPIO_ID_CTRX_RESET_N, true); // Release reset PlatformTime_waitTime(10);

Step 8: Initialize CTRX_A (Primary Device) (main.c:192-207)

8.1 Prepare Primary Configuration

IRAM descriptor values configure CTRX_A as PRIMARY using LOIN2:

const uint32_t intitializeValuePrimary[] = { 15, // Primary mode using Loin2 2, // Disable CRC32 0, // 1 data packet 0, // Mode a 0 // Non-continuous clock mode };

8.2 Call IfxRfe_ctrxInit

IfxRfe_ctrxInitConfiguration_t ctrxInitConfig = prepareCtrxInitConfig(PRIMARY); ctrxInitConfig.spiClockSpeed = 3125000; // 3.125 MHz ctrxInitConfig.initializeConfig.iram_descriptor = intitializeIndex; ctrxInitConfig.initializeConfig.value = intitializeValuePrimary; IfxRfe_selectDevice(0); // Select CTRX_A IfxRfe_ctrxInit(ctrxInitConfig);

IfxRfe_ctrxInit performs:

  • Download RAM firmware to CTRX_A
  • Configure MMIC clock settings
  • Execute initialization with PRIMARY mode parameters
  • Wait for RFT pin to go HIGH (indicates firmware boot success)
  • Transition to low-power state

Step 9: Initialize CTRX_B (Secondary Device) (main.c:209-216)

CTRX_B Configuration as Secondary Using LOIN1:

const uint32_t intitializeValueSecondary_B[] = { 4, // Secondary mode using Loin1 2, // Disable CRC32 0, // 1 data packet 0, // Mode a 0 // Non-continuous clock mode }; ctrxInitConfig = prepareCtrxInitConfig(SECONDARY); ctrxInitConfig.spiClockSpeed = 3125000; ctrxInitConfig.initializeConfig.value = intitializeValueSecondary_B; IfxRfe_selectDevice(1); // Select CTRX_B IfxRfe_ctrxInit(ctrxInitConfig);

CTRX_B synchronizes to CTRX_A via LOIN1 signal for phase-coherent operation.

Step 10: Initialize CTRX_C (Secondary Device) (main.c:218-225)

CTRX_C Configuration as Secondary Using LOIN2:

const uint32_t intitializeValueSecondary_C[] = { 12, // Secondary mode using Loin2 2, // Disable CRC32 0, // 1 data packet 0, // Mode c 0 // Non-continuous clock mode }; ctrxInitConfig = prepareCtrxInitConfig(SECONDARY); ctrxInitConfig.spiClockSpeed = 3125000; ctrxInitConfig.initializeConfig.value = intitializeValueSecondary_C; IfxRfe_selectDevice(2); // Select CTRX_C IfxRfe_ctrxInit(ctrxInitConfig);

CTRX_C synchronizes to CTRX_A via LOIN2 signal (mode c configuration).

Step 11: Load Sequencer Programs (main.c:228-237)

Sequencer Program Differences:

Primary and secondary devices use different sequencer programs to coordinate synchronized operation:

11.1 Load Primary Sequencer (CTRX_A):

const uint32_t sequencer_program_data_primary[] = { // Primary sequencer with trigger enable flags set 0x33880024, 0x00000000, 0x01000100, 0x01000100, ... }; IfxRfe_selectDevice(0); IfxRfe_loadSequencerData(0, sequencer_program_data_primary, length);

11.2 Load Secondary Sequencer (CTRX_B and CTRX_C):

const uint32_t sequencer_program_data_secondary[] = { // Secondary sequencer - responds to primary triggers 0x33880024, 0x00000000, 0x01000100, 0x01000100, ... }; IfxRfe_selectDevice(1); IfxRfe_loadSequencerData(0, sequencer_program_data_secondary, length); IfxRfe_selectDevice(2); IfxRfe_loadSequencerData(0, sequencer_program_data_secondary, length);
Sequencer Details: The sequencer programs define 512 ramps × 512 samples × 8 RX channels per chip. The primary device's sequencer includes trigger enable flags that secondary devices listen for via LOIN synchronization.

Step 12: Configure DMUX for Synchronization (main.c:240-242)

Configure DMUX_1 on CTRX_A (primary) as output to signal ramp completion:

IfxRfe_selectDevice(0); IfxRfe_configureDmux(_configParams.dmux);

DMUX_1_A configuration from config.c:54-61:

  • config_mask: ConfigMask_DMUX1 (update only DMUX_1)
  • dmux1_alt_signal: AltSignal_DmuxALevel (map level, not pulse)
  • dmux1_dir: DmuxDir_out (output)

This DMUX signal is used by the host to detect when the ramp scenario completes.

Step 13: Configure All Three Chips (main.c:245-287)

Loop through all three devices (i=0,1,2) and configure each with identical radar parameters:

For each CTRX (A, B, C):

13.1 Clear Results
IfxRfe_clearResults(_configParams.clearResults);
13.2 Configure Ramp Scenario
IfxRfe_configureRampScenario(_configParams.rampScenario);

Sets sequencer start address to 0.

13.3 Configure RF Frequency (from config.c)
IfxRfe_configureRfFrequency(_configParams.rfFreq);

RF Parameters:

  • nmod: 62432208 (static frequency ~79.65 GHz)
  • ncw: 283073584 (lock frequency ~80.988 GHz)
  • rampbw: 8738128 (~2.5 GHz bandwidth)
  • bc: 1 (bandwidth control)
13.4 Configure TX Power
IfxRfe_configureTxPower(_configParams.txpwr);

All TX channels set to power level 0 (configurable per channel).

13.5 Configure RX Frontend
IfxRfe_configureRx(_configParams.rxcfg);

RX Configuration:

  • Gain: 0 dB
  • Data Width: 12 bits
  • Data Rate: 1200 Mbps per CSI-2 lane
  • Start Mode: Immediate
13.6 Read Temperature
IfxRfe_getTemperature(&tempRes); printf(" Temperature Sensor 1: %.1f °C\n", tempRes.temp1 / 8.0f); printf(" Temperature Sensor 2: %.1f °C\n", tempRes.temp2 / 8.0f);
13.7 Execute RX Calibration
IfxRfe_executeCalibration(_configParams.calibration, NULL);

Calibration sub-functions: 0x1B00 (RX calibrations)

13.8 Go to Operation Mode
IfxRfe_gotoOperation(); IfxRfe_getStatus(&status);

Transitions chip from low-power to operational state. Verifies state is 0x10000000 (operational).

Step 14: Execute TX Calibration with Asynchronous Start (main.c:290-302)

TX calibration is performed across all three chips using asynchronous parallel execution:

// Set TX calibration sub-functions (all TX calibrations) _configParams.calibration.calib_sub_func_id = 0x000F; // Start CTRX_C calibration (non-blocking) IfxRfe_selectDevice(2); IfxRfe_executeCalibration_start(_configParams.calibration, &handle_C); // Start CTRX_B calibration (non-blocking) IfxRfe_selectDevice(1); IfxRfe_executeCalibration_start(_configParams.calibration, &handle_B); // Execute CTRX_A calibration (blocking) IfxRfe_selectDevice(0); IfxRfe_executeCalibration(_configParams.calibration, NULL); // Wait for CTRX_C to finish IfxRfe_selectDevice(2); IfxRfe_executeCalibration_finish(handle_C, NULL); // Wait for CTRX_B to finish IfxRfe_selectDevice(1); IfxRfe_executeCalibration_finish(handle_B, NULL);

This pattern allows TX calibration to run concurrently on all three chips, reducing total calibration time.

Step 15: Wait for DMUX Ready State (main.c:305)

PlatformGpio_gpioWait(GPIO_ID_DMUX_1_A, false, 10);

Wait for DMUX_1_A to go LOW (indicating primary is ready to start a new ramp). Timeout of 10ms.

Step 16: Interactive Operation Loop (main.c:307-337)

Enter an interactive loop for up to 10,000 iterations:

16.1 User Prompt

printf("Enter 'c' to continue or 'e' to exit: "); scanf(" %c", &user_input); if (user_input == 'e') break;

16.2 Read Temperature (CTRX_A)

IfxRfe_getTemperature(&tempRes); printf(" Temperature Sensor 1: %.1f °C\n", tempRes.temp1 / 8.0f);

16.3 Start Ramp Scenario on All Chips (Reverse Order)

Critical: Start secondary devices first, then primary to ensure synchronization:

for (int i = 2; i >= 0; i--) // C, B, A order { IfxRfe_selectDevice(i); IfxRfe_startRampScenario(); }
Synchronization Strategy: Secondary devices (C and B) are started first and wait for synchronization signals from the primary (A). The primary device is started last and triggers all devices to execute the ramp scenario simultaneously via LOIN signals.

16.4 Wait for Ramp Completion

PlatformGpio_gpioWait(GPIO_ID_DMUX_1_A, true, 15);

Wait for DMUX_1_A to go HIGH (ramp scenario executing). Timeout of 15ms.

16.5 Finish Ramp Scenario on All Chips

for (int i = 0; i < 3; i++) // A, B, C order { IfxRfe_selectDevice(i); IfxRfe_finishRampScenario(NULL); }

Call finishRampScenario to complete the operation and return chips to ready state.

Step 17: Cleanup and Exit (main.c:340-343)

Upon exit or error, the cleanup function executes:

void cleanup_platform() { PlatformGpio_set(GPIO_ID_CTRX_RESET_N, false); // Reset all CTRXs PlatformI2c_deinit(); PlatformSpi_deinit(); PlatformGpio_deinit(); } cleanup_platform(); printf(" === SUCCESS ===\n"); return EXIT_SUCCESS;

3. Key Components and Their Roles

3.1 main.c

Role: Application entry point and orchestration

  • Initializes platform (GPIO, SPI, I2C)
  • Manages power-up sequence for PMICs and radar chips
  • Coordinates initialization of three CTRX devices in PRIMARY/SECONDARY modes
  • Loads device-specific sequencer programs
  • Orchestrates synchronized radar operation across all three chips
  • Provides interactive user interface for radar control

Key Functions:

  • main() - Entry point with complete initialization and operation flow
  • cleanup_platform() - Cleanup resources on exit

3.2 config.c / config.h

Role: Radar configuration parameters

  • Defines RF frequency parameters (79.65-80.988 GHz, 2.5 GHz BW)
  • Configures TX power levels (all channels at 0 dB)
  • Sets RX frontend parameters (0 dB gain, 12-bit data, 1200 Mbps)
  • Defines calibration parameters (RX and TX calibration sub-functions)
  • Configures DMUX for ramp completion signaling

Key Structure: IfxRfe_demoConfigParams_t - Container for all configuration parameters

Key Function: IrfeDemoConfigInit() - Initializes configuration structure

3.3 HwDefinitions.h

Role: Hardware pin mapping definitions

  • Maps Jetson Orin GPIO port/pin offsets to absolute GPIO numbers
  • Defines GPIO IDs for all three CTRX chips (SPI CS, RFT, DMUX pins)
  • Defines shared control signals (PMIC enable, reset, power good)
  • Maps SPI device IDs (0x10, 0x11, 0x12)
  • Defines I2C bus ID (I2C3 = bus ID 2)

3.4 IfxRfe Middleware (submodules/iRfe/)

Role: Infineon Radio Frequency Engine API

  • Provides high-level radar control API
  • Handles SPI command protocol for CTRX8188 chips
  • Manages firmware download and initialization
  • Implements synchronous and asynchronous command execution
  • Supports multi-device management with device selection

Key Functions:

  • IfxRfe_init() - Initialize middleware for multiple devices
  • IfxRfe_selectDevice() - Select active device for commands
  • IfxRfe_ctrxInit() - Initialize CTRX chip (firmware, clock, state)
  • IfxRfe_loadSequencerData() - Load sequencer program
  • IfxRfe_startRampScenario() - Start radar chirp sequence
  • IfxRfe_finishRampScenario() - Complete radar operation
  • IfxRfe_executeCalibration_start() - Non-blocking calibration
  • IfxRfe_executeCalibration_finish() - Wait for calibration completion

3.5 IfxRfe_Specific.c/h (8188 chip-specific)

Role: CTRX8188-specific initialization and firmware management

  • prepareCtrxInitConfig() - Prepares init config for PRIMARY/SECONDARY/STANDALONE modes
  • IfxRfe_loadRamFw() - Downloads RAM firmware frames to chip
  • Defines chip-specific initialization parameters
  • Handles firmware download protocol and error clearing

3.6 Platform Layer (submodules/platform-jetson/)

Role: Hardware abstraction for Jetson Orin platform

  • PlatformGpio: GPIO control via libgpiod v1.6
    • configure() - Set GPIO direction and initial state
    • get() / set() - Read/write GPIO values
    • gpioWait() - Wait for GPIO state change with timeout
  • PlatformSpi: SPI communication via spidev
    • Manual CS control for multi-frame transactions
    • 3.125 MHz clock speed
    • Support for multiple device IDs with GPIO CS mapping
  • PlatformI2c: I2C communication via SMBus
    • Used for PMIC control
  • PlatformTime: Time management (delays, timeouts)

3.7 3xCtrx8188Wrapper.c

Role: Board-specific SPI transfer workarounds

  • PlatformSpi_transfer_override() - Wraps standard transfer with delays
  • PlatformSpi_write_override() - Wraps standard write with delays
  • Adds 2ms delay after trigger_reset, trigger_LBIST, and finish_fw_download commands
  • Prevents RFT pin from being read before stabilization (EK1 board-specific issue)

3.8 Max20434Pec.c (pmic_power/)

Role: MAX20434 PMIC control driver

  • Max20434_enableSpreadSpectrum() - Configure spread spectrum mode
  • Max20434_setManualOutputEnable() - Enable/disable PMIC outputs
  • Communicates via I2C at addresses 0x3E and 0x3F

3.9 capture_all_8x512x512.sh

Role: Parallel data capture script for three radar streams

  • Captures radar data from three V4L2 video devices simultaneously
  • Configures format: RAW12, 4096×512 pixels (8 RX channels × 512 samples × 512 ramps)
  • Outputs: ctrx0_bin.raw, ctrx1_bin.raw, ctrx2_bin.raw
  • Runs three v4l2-ctl instances in parallel
  • Must be executed after radar enters operational mode

6. Synchronization and Cascading Architecture

6.1 Primary-Secondary Synchronization

How Cascading Works:

  • CTRX_A (Primary): Generates local oscillator signals on LOIN1 and LOIN2 outputs
  • CTRX_B (Secondary): Receives clock from CTRX_A via LOIN1 input
  • CTRX_C (Secondary): Receives clock from CTRX_A via LOIN2 input
  • Phase Coherence: All three chips share the same LO signal, enabling coherent MIMO radar processing

6.2 Initialization Mode Configuration

Device Mode LOIN Config Value Synchronization
CTRX_A PRIMARY 15 (LOIN2 master) Generates LO for B and C
CTRX_B SECONDARY 4 (LOIN1 slave) Locked to CTRX_A LOIN1
CTRX_C SECONDARY 12 (LOIN2 slave) Locked to CTRX_A LOIN2

6.3 Sequencer Program Differences

Primary Sequencer: Contains trigger enable flags that secondary devices monitor. When primary starts a ramp, it sends synchronization signals via LOIN that secondary devices detect and respond to.

Secondary Sequencer: Configured to wait for external trigger signals. When LOIN signals indicate primary has started, secondary devices execute their sequencer programs in lockstep.

9. Critical Timing and Synchronization Constraints

9.1 Power-Up Sequence

  • CTRX_RESET_N must be LOW during PMIC power-up (prevents GPIO leakage issues)
  • Minimum 10ms delay after PMIC enable before releasing reset
  • PMIC output reset workaround: 100ms disable period required

9.2 SPI Communication

  • Maximum SPI clock speed: 3.125 MHz (higher speeds cause communication failures)
  • 2ms delay required after reset/LBIST/finish_fw_download commands for RFT stabilization
  • Manual CS control required to maintain CS assertion across multi-frame transfers

9.3 Cascaded Ramp Execution

  • Secondary devices must be armed BEFORE primary device (reverse order: C→B→A)
  • Primary device triggers all devices simultaneously via LOIN signals
  • DMUX_1_A LOW indicates ready state (timeout 10ms)
  • DMUX_1_A HIGH indicates ramp executing (timeout 15ms)
  • All devices must finish in forward order (A→B→C) for proper cleanup

9.4 CSI-2 Data Capture

  • Capture must start AFTER radar enters operational mode
  • CSI-2 receiver requires active clock from radar to enable lane termination
  • Three captures must run in parallel to avoid data loss
  • Continuous clock mode disabled (controlled by sequencer program)

10. Error Handling and Macros

The application uses two primary error-handling macros defined in error_macros.h:

EXIT_ON_PLATFORM_ERROR(function_call, cleanup_action) - Checks return value of platform functions - Executes cleanup_action and exits if error occurs - Used for GPIO, SPI, I2C initialization and operations EXIT_ON_IFXRFE_ERROR(function_call, cleanup_action) - Checks return value of IfxRfe API functions - Executes cleanup_action and exits if error occurs - Used for all radar chip configuration and control commands

Cleanup Strategy: All critical sections use these macros to ensure proper resource cleanup on failure. The cleanup_platform() function resets radar chips and deinitializes all platform resources.

12. Summary

This cascaded 3x CTRX8188 radar application demonstrates a sophisticated multi-chip MIMO radar system with the following key characteristics:

  • Scalable Architecture: Extends from single-chip to three-chip operation with primary-secondary synchronization
  • Phase-Coherent Operation: LOIN signals ensure all three chips operate with locked local oscillators for coherent radar processing
  • Parallel Processing: Asynchronous calibration and parallel data streaming maximize system throughput
  • Robust Error Handling: Comprehensive cleanup on all error paths prevents resource leaks
  • Hardware Workarounds: Includes board-specific fixes for PMIC output reset and SPI timing issues
  • Interactive Operation: User-controlled radar chirp execution with temperature monitoring
  • Triple Data Streams: Simultaneous capture from three independent CSI-2 interfaces totaling 24 RX channels

Total Pin Count: 23 GPIOs + 3 SPI shared + 3 SPI CS + 2 I2C + 15 CSI-2 lanes = 46 interconnects between Jetson Orin and radar board via J509 camera connector.

Future Enhancement: This system is designed to integrate GMSL SERDES links for long-distance radar data transmission, enabling remote radar sensor deployments while maintaining phase-coherent operation across multiple radar arrays.
